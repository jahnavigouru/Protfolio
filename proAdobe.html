    • Internship

    Part of Adobe campaign's UCO team. UCO, the "Unified Campaign Operations" is a self-servicing platform that unifies all the operational-oriented tools required for the Adobe Campaign. The first 3-weeks of the 6-month long internship were the learning phase, which involved implementing key-value as a service using golang, react, MongoDB, and docker for dockerizing all the services.

    This internship revolved around exploring, designing, and developing internal tools for the UCO to improve the consistency and reliability of campaign infrastructure and speed up the value delivery of rollouts on Adobe campaign.

    Features

    • Scheduling Assistant
    • Restart Manager

    The Problem

    Adobe campaign requires numerous continuous deployments to avoid these rollouts getting scheduled in the same time frame and conflicting with any pre-existing rollouts, the UCO team needed a way to provide PgMs more visibility into the rollout schedule.

    At present, it was done by some intermediary which isn't feasible given the wide range of rollout adoption.

    Business value:
  • Help PgMs plan better rollouts.
  • Better UX in rollout planning

    • The Solution

    Scheduling Assistant
  • This provides one single view with all the scheduled releases of the products (ACC / ACS ) in the form of a calendar.
  • This information helps in effectively scheduling releases.
  • Color coding is used to indicate if the slots are exceeding the product's execution capacity so that the user stays warned while scheduling a new release.
  • Provision to show change restricted slots to align with Adobe wide change restrictions as well as internal UCO specific releases.

    • Feature Implementation Process

    After understanding UCO's complex architecture, the problem statement, and the impact of scheduling assistant as a feature.

    Execution Flow
  • Phase 1: Documenting and prototyping different ideas for implementing scheduling assistant and getting them reviewed by the UCO team.
  • Phase 2: Writing an effective psql query for fetching data of all the rollouts scheduled in a specified week from the database. Three iterations were required for optimizing the time complexity of the algorithm for intelligently showing rollouts in the calendar which was followed by server-side API implantation.
  • Phase 3: Implementation of UI component and API integration for the assistant using react+typescript.
  • Phase 4: Testing different edge cases and fixing them.
  • Phase 5: Launching it to the production where PgMs could finally make use of the feature.

    • Feature Jounery

    The entire feature took an active two and a half months (approx.) to complete, starting from February after finishing the Warmup Project.The data is stored in PostgreSQL, the backend is written in Golang and the frontend is in react + typescript, whereas for a weekly view of the calendar we used an open-source dependency called full calendar, and the rest of the components use react-spectrum which is Adobe's design system. And the services were dockerized.

    The Problem

    For business continuity without a fail, specifically for a sprawling infrastructure like Adobe Campaign's, it's important to eliminate manual processes with self-service tooling. So, to ensure the infrastructure adheres to standards without an exception, we need an "automate everything" approach.

    Business value:
  • Reduced human errors.
  • Better customer experience.

    • The Solution

    Restart manager

    Automate the end-to-end process of Campings's clean restart.
    Restart manager is implemented using airflow DAG. Apache Airflow is a workflow management platform for data engineering pipelines and in Airflow, a DAG or a Directed Acyclic Graph - is a collection of all the tasks you want to run, organized in a way that reflects their relationships and dependencies. Airflow is written in Python, and workflows are created via Python scripts.

  • Restart Manager Dag performs tasks like pre/post checks, mute/unmute alerts, and supports strategies like rolling/parallel.

  • This Dag executes these tasks conditionally by calling different airflow operators (like BashOperator and BranchPythonOperator) with different operator arguments for executing the change.

    • Flash Operator

    The Flash operator is a generic airflow operator that provides an interface for executing any salt modules with custom parameters.This submits the job using flash API, where the entire job execution process is going to be asynchronous. Later for checking the job execution status, a robust polling mechanism is followed while exponentially backing off on certain HTTP codes like 429(Too Many Requests), 500(Internal Server Error), 502(Bad Gateway), 504(Gateway Timeout) Parallel/Rolling restart task mentioned above calls Flash Operator for performing a restart on the list of servers provided by the user.
